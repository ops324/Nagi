## 1. プロジェクト概要

### 1.1 アプリ名
**Nagi（凪）** — 自己観察の記録

### 1.2 コンセプト
日記・ジャーナル形式で今の気持ちや思考を記録すると、AIキャラクター「凪」が評価・判断をせず、静かな問いかけでコメントを返すWebアプリ。

凪の役割は2つ。ひとつは出来事の表面だけでなく構造・前提への気づきを促す**本質的な理解把握への伴走**。もうひとつは今使っている視点の外側に扉が存在することを示す**多面的視点による次元上昇**。自己理解が深まることで、他者理解へも自然につながっていく体験を目指す。

### 1.3 デザイン哲学
**Calm Technology × Soft Minimalism**

> 「生産性ツールではなく、静かな日記帳のように感じられるインターフェース」

- **Calm Technology** — 主張せず、そっと寄り添うUI
- **Human-Centered Design** — ユーザーの内省を邪魔しない
- **Soft Minimalism** — 余白と柔らかさで安心感を生む

### 1.4 ターゲットユーザー
- 自己理解を深めたい人
- 感情や思考を整理したい人
- 日記・ジャーナリングを習慣にしたい人

### 1.5 AIキャラクター設定（凪）
| 項目 | 内容 |
|------|------|
| 名前 | 凪（Nagi） |
| 役割 | 自己観察の静かな伴走者 |
| トーン | 温かく、淡々とした語り口 |
| 方針 | 評価・判断をしない。問いかけや観察を通じて気づきを促す |
| コメント文字数 | 入力長に応じた段階制（短文30〜50字／中文60〜100字／長文100〜150字程度。v1.40.0 参照） |

### 5.1 POST `/api/comment`

AIコメント生成＋感情抽出エンドポイント（**認証必須**・サーバーサイド処理）。

**認証:** Supabase セッションクッキー必須
**レート制限:** 1ユーザーあたり 20回/時間
**入力制限:** 最大5000文字

**リクエスト**
```json
{ "content": "ユーザーが入力した記録テキスト" }
```

**レスポンス（成功）**
```json
{
  "comment": "凪からのコメントテキスト（入力長に応じた段階制。v1.40.0 参照）",
  "emotions": [{ "label": "喜び", "score": 0.7 }],
  "dominant": "喜び",
  "energy": 5,
  "insightLevel": "moderate"
}
```

**使用モデル:** `claude-haiku-4-5-20251001` / **max_tokens:** 入力長ティアで 300/500/800 / **system:** 4モード応答プロンプト v2.0

**リトライ／エラー:** Anthropic の一時エラー（429/5xx/529 Overloaded・接続エラー）は指数バックオフで再試行（`lib/anthropic-retry.ts`）。再試行後も過負荷なら **503** ＋ `{ "error": "ただいま混み合っています。少し時間をおいてもう一度お試しください" }`。それ以外の生成失敗は **500** ＋ `{ "error": "コメントの生成に失敗しました" }`。AI 出力の JSON パース失敗時は 500 にせず各フィールドの既定値でフォールバックする。

### 5.3 DELETE `/api/account/delete`

アカウント完全削除エンドポイント（認証必須・サービスロールキー使用）。

**認証:** Supabase セッションクッキー必須
**レート制限:** 1ユーザーあたり 3回/時間

**処理順序:**
1. セッションからユーザーIDを取得・認証確認
2. `entries` テーブルから当該ユーザーの全記録を削除
3. `profiles` テーブルから当該ユーザーのプロフィールを削除
4. `auth.admin.deleteUser()` でAuthユーザーを削除

**レスポンス（成功）**
```json
{ "success": true }
```

**レスポンス（エラー）**
```json
{ "error": "Unauthorized" }       // 401: 未認証
{ "error": "Failed to delete account" }  // 500: 削除失敗
```

### 5.4 POST `/api/comment/demo`

登録前のお試し体験（`/try`）用のコメント生成エンドポイント（**認証なし**・サーバーサイド処理・v1.59.0）。生成ロジック本体は `/api/comment` と共通（`lib/generate-comment.ts`）。

**認証:** 不要（公開ルート。`publicPaths` に登録）
**レート制限:** **IP単位** 5回/時間（乱用・コストのバックストップ。共有IPの正当な利用者を巻き込まないやや緩めの値）
**入力制限:** 最大2000文字（コスト抑制のため本番より短い）
**保存:** なし（呼び出し側 `DemoClient` で DB・localStorage に保存しない）

**リクエスト／レスポンス:** `/api/comment` と同一形（`comment` / `emotions` / `dominant` / `energy` / `insightLevel`）。
**エラー:** 上限超過時 **429** ＋ `{ "error": "お試しの上限に達しました。登録すると、続けてご利用いただけます" }`。一時エラー時 503・その他 500 は `/api/comment` と同方針。

> ※ 体験を「1回だけ」に絞るのは `/try` の UI 側の責務。本APIのIP制限は乱用防止が目的。

```typescript
  user_id: string;      // Supabase auth.users.id（v1.17追加）
  content: string;
  comment: string;
  emotions: Emotion[];
  dominant: string;
  energy: number;       // 1〜10
  createdAt: string;    // ISO 8601
  insightLevel?: "deep" | "moderate" | "gentle";  // 洞察レベル（v1.23追加）
};
```

### 6.2 DBスキーマ（Supabase PostgreSQL）

```sql
-- ユーザープロフィール
public.profiles (
  id uuid PRIMARY KEY,  -- auth.users.id と同一
  email text,
  is_admin boolean DEFAULT false,
  created_at timestamptz
)

-- 日記記録
public.entries (
  id text PRIMARY KEY,
  user_id uuid REFERENCES profiles(id),
  content text,
  comment text,
  emotions jsonb,
  dominant text,
  energy integer,
  created_at timestamptz,
  insight_level text DEFAULT 'moderate',  -- "deep" | "moderate" | "gentle"
  note text                               -- 余韻メモ（nullable）

-- 管理者用ビュー
public.admin_analytics     -- ユーザー別集計（総記録数・最終記録・平均エネルギー）
public.admin_emotion_stats -- 感情ラベル別集計
```

| テーブル | 操作 | ポリシー |
|---------|------|---------|
| profiles | SELECT / UPDATE | 自分のレコードのみ（`auth.uid() = id`） |
| entries | SELECT / INSERT / UPDATE / DELETE | 自分のレコードのみ（`auth.uid() = user_id`） |

## 7. 認証フロー（A-1 移行後・v1.32）

```
リクエスト
    │
    ▼
proxy.ts（トークンリフレッシュのみ）
    │ lib/supabase/middleware.ts の updateSession を呼ぶ
    │ getClaims() でローカル JWT 検証 → 必要に応じてリフレッシュ
    │ 認証ガード・リダイレクトは行わない
    ▼
各ページ / Route Handler（それぞれが getClaims() でガード）
    │ /auth/login → 認証済みなら / にリダイレクト
    │ / → Server Component で getClaims() を実行
    │       認証エラー or claims 未取得 → redirect("/auth/login")
    │ /account → 未認証なら /auth/login にリダイレクト
    │ /admin → layout.tsx で認証 + is_admin チェック、未認証/非管理者はリダイレクト
    │ /api/* → getClaims() エラーなら 401 を返す
    ▼
/ メインページ
    │ Supabase DBからentriesを取得（自分のデータのみ）
    │ 記録0件：ウェルカム画面を全画面表示（「はじめる」で閉じてホーム画面へ）
    ▼
ヘッダー
    └── is_admin=true → /admin リンク表示
    └── ログアウトボタン → /auth/login へ
```

**公開ルート（未認証で通過・`lib/supabase/middleware.ts` の `publicPaths`）:** `/privacy`・`/terms`・`/auth/callback`・`/try`（登録前のお試し体験・v1.59.0）・`/api/comment/demo`（お試し用コメント生成・v1.59.0）。

### 認証 API 早見表

| シーン | 推奨 API | 理由 |
|--------|----------|------|
| ページ／API での認証チェック | `supabase.auth.getClaims()` | ローカル検証で高速（ES256） |

### app/page.tsx の認証ガード実装詳細（v1.56.0 で Server Component 化）

`app/page.tsx` は **Server Component**。サーバー側で `getClaims()` を1回実行し、認証エラーまたは claims 未取得時に `redirect("/auth/login")` する。あわせてプロフィール（`is_admin`）と記録（`entries`）をサーバーで取得し、`app/components/HomeClient.tsx`（Client Component）へ props として渡す。HomeClient は受け取った初期データで state を初期化し、対話処理（作成・編集・削除・お気に入り・余韻メモ等）を担う。

```typescript
// app/page.tsx（Server Component）
const supabase = await createClient();
const { data: claimsData, error } = await supabase.auth.getClaims();
if (error || !claimsData?.claims) redirect("/auth/login");
const userId = claimsData.claims.sub;

const [{ data: profile }, { data }] = await Promise.all([
  supabase.from("profiles").select("is_admin").eq("id", userId).single(),
  supabase.from("entries").select("*").eq("user_id", userId).order("created_at", { ascending: false }),
]);
// initialEntries にマッピングして <HomeClient initialEntries=... userEmail=... isAdmin=... />
```

### 管理者アナリティクスのRLS設定

管理者が全ユーザーデータを参照するために、以下のSQL（`docs/supabase-setup.sql` に含む）を実行が必要：

```sql
-- 管理者権限チェック関数（SECURITY DEFINER でRLSをバイパス）
CREATE OR REPLACE FUNCTION public.is_admin()
RETURNS boolean AS $$
  SELECT COALESCE(
    (SELECT is_admin FROM public.profiles WHERE id = auth.uid()),
    false
  )
$$ LANGUAGE sql SECURITY DEFINER;

-- 管理者は全プロフィール・全エントリを参照可
CREATE POLICY "管理者は全プロフィール参照可" ON public.profiles FOR SELECT
  USING (auth.uid() = id OR public.is_admin());

CREATE POLICY "管理者は全エントリ参照可" ON public.entries FOR SELECT
  USING (auth.uid() = user_id OR public.is_admin());
```

## 11. セキュリティ

| 項目 | 対応 |
|------|------|
| APIキー漏洩防止 | `.env.local` は `.gitignore` で Git 管理外 |

### 13.1 設計概要

7つの設計思想カテゴリ（量子力学的観測・言語化蓄積・戦略的自己規律・物語贈与余白・ナラティブエンジニアリング・フィルム理論・マインドフルネス効果）を、ユーザーの状態に応じて切り替わる **4つの応答モード** として統合しています。

| モード | 発動条件 | 思想源泉 | トーン |
|--------|----------|---------|--------|
| A: 受容 | 辛い・疲弊・混乱 | Cat.1,4,7 | 深い静けさ・温もり |
| B: 観測 | 日常・中立・淡々 | Cat.1,2,6,7 | 穏やかな好奇心 |

### 13.2 AIコメント基本原則

| 原則 | 内容 |
|------|------|
| 非評価 | 「良い／悪い」の判断を下さない |
| 観察促進 | 気づきを深めるための問いかけ |
| 重ね合わせの肯定 | 複数感情の同居をそのまま肯定 |
| 物語の連続性 | 単発の記録を自己の物語の一点として位置づける |

### 14.3 CI（GitHub Actions）

`.github/workflows/ci.yml`。PR と main への push で実行。

| ジョブ | 手順 |

### 14.4 既知の留意点・運用ルール

- **ESLint ignore**: `.claude/**`・`supabase/**`・`test-results/**`・`coverage/**` を除外。これがないと `npm run lint` がローカルの `.claude/worktrees/`（プロジェクト複製）を走査し大量の問題を誤報告する。
- **react-hooks v6 の新規ルール**: `react-hooks/set-state-in-effect`・`react-hooks/purity` を一時的に `warn` に緩和（`HomeClient.tsx`・`DemoClient.tsx` の既存パターン）。実コード修正はフェーズ3（HomeClient 分割）で対応予定。
- **package-lock.json の同期**: `npm install` 直後は optional な wasm 依存（`@emnapi/*`）や vite 版の衝突で lock がドリフトし CI の `npm ci` が失敗することがある。依存追加後は `rm -rf node_modules package-lock.json && npm install` でクリーン再生成し、`npm ci` が通ることを確認してから push する。vitest@4 と `@sentry/nextjs` の vite はともに `^6` を受理するため `package.json` の `overrides: { "vite": "^6.0.0" }` で単一版に固定して解消している。
