## 1. プロジェクト概要

### 1.1 アプリ名
**Nagi（凪）** — 自己観察の記録

### 1.2 コンセプト
日記・ジャーナル形式で今の気持ちや思考を記録すると、AIキャラクター「凪」が評価・判断をせず、静かな問いかけでコメントを返すWebアプリ。

凪の役割は2つ。ひとつは出来事の表面だけでなく構造・前提への気づきを促す**本質的な理解把握への伴走**。もうひとつは今使っている視点の外側に扉が存在することを示す**多面的視点による次元上昇**。自己理解が深まることで、他者理解へも自然につながっていく体験を目指す。

### 1.3 デザイン哲学
**Calm Technology × Soft Minimalism**

> 「生産性ツールではなく、静かな日記帳のように感じられるインターフェース」

- **Calm Technology** — 主張せず、そっと寄り添うUI
- **Human-Centered Design** — ユーザーの内省を邪魔しない
- **Soft Minimalism** — 余白と柔らかさで安心感を生む

### 1.4 ターゲットユーザー
- 自己理解を深めたい人
- 感情や思考を整理したい人
- 日記・ジャーナリングを習慣にしたい人

### 1.5 AIキャラクター設定（凪）
| 項目 | 内容 |
|------|------|
| 名前 | 凪（Nagi） |
| 役割 | 自己観察の静かな伴走者 |
| トーン | 温かく、淡々とした語り口 |
| 方針 | 評価・判断をしない。問いかけや観察を通じて気づきを促す |
| コメント文字数 | 100〜150文字程度 |

### 5.2 DELETE `/api/account/delete`

アカウント完全削除エンドポイント（認証必須・サービスロールキー使用）。

**認証:** Supabase セッションクッキー必須
**レート制限:** 1ユーザーあたり 3回/時間

**処理順序:**
1. セッションからユーザーIDを取得・認証確認
2. `entries` テーブルから当該ユーザーの全記録を削除
3. `profiles` テーブルから当該ユーザーのプロフィールを削除
4. `auth.admin.deleteUser()` でAuthユーザーを削除

**レスポンス（成功）**
```json
{ "success": true }
```

**レスポンス（エラー）**
```json
{ "error": "Unauthorized" }       // 401: 未認証
{ "error": "Failed to delete account" }  // 500: 削除失敗
```

  email text,

## 7. 認証フロー（A-1 移行後・v1.32）

```
リクエスト
    │
    ▼
proxy.ts（トークンリフレッシュのみ）
    │ lib/supabase/middleware.ts の updateSession を呼ぶ
    │ getClaims() でローカル JWT 検証 → 必要に応じてリフレッシュ
    │ 認証ガード・リダイレクトは行わない
    ▼
各ページ / Route Handler（それぞれが getClaims() でガード）
    │ /auth/login → 認証済みなら / にリダイレクト
    │ / → useEffect 内で getClaims() を実行（Client Component）
    │       認証エラー or claims 未取得 → router.replace("/auth/login")
    │ /account → 未認証なら /auth/login にリダイレクト
    │ /admin → layout.tsx で認証 + is_admin チェック、未認証/非管理者はリダイレクト
    │ /api/* → getClaims() エラーなら 401 を返す
    ▼
/ メインページ
    │ Supabase DBからentriesを取得（自分のデータのみ）
    │ 初回ログイン時：空の状態で表示（「まだ記録がありません」）
    ▼
ヘッダー
    └── is_admin=true → /admin リンク表示
    └── ログアウトボタン → /auth/login へ
```

### 認証 API 早見表

| シーン | 推奨 API | 理由 |
|--------|----------|------|
| ページ／API での認証チェック | `supabase.auth.getClaims()` | ローカル検証で高速（ES256） |
| ユーザー情報の即時更新が必要 | `supabase.auth.getUser()` | サーバーから最新情報を取得 |
| ログイン状態の変更通知 | `supabase.auth.onAuthStateChange()` | リアルタイム |

### app/page.tsx の認証ガード実装詳細

`app/page.tsx` は Client Component のまま維持。`useEffect` 内の `loadUser` / `loadEntries` それぞれで `getClaims()` を呼び出し、認証エラーまたは claims 未取得時に `router.replace("/auth/login")` でリダイレクトする。

```typescript
const loadUser = async () => {
  const { data: claimsData, error } = await supabase.auth.getClaims();
  if (error || !claimsData?.claims) { router.replace("/auth/login"); return; }
  // ...
};

const loadEntries = async () => {
  const { data: claimsData, error } = await supabase.auth.getClaims();
  if (error || !claimsData?.claims) { router.replace("/auth/login"); return; }
  // ...
};
```

### 既知の制約・将来課題

- `app/page.tsx` は Client Component のため、コンポーネントマウント後に認証チェックが走る。未ログインアクセス時にマウント時点で一瞬コンテンツが見える可能性あり。将来的に Server Component 化で解消を検討
- `handleSubmit()` 内の `entry.id = Date.now().toString()` は理論上の衝突リスクあり（将来 `crypto.randomUUID()` への置換を別タスクで検討）

### 管理者権限の付与方法

Supabase SQL Editorで実行：
```sql
UPDATE public.profiles SET is_admin = true WHERE email = '管理者のメールアドレス';
```

### 管理者アナリティクスのRLS設定

管理者が全ユーザーデータを参照するために、以下のSQL（`docs/supabase-setup.sql` に含む）を実行が必要：

```sql
-- 管理者権限チェック関数（SECURITY DEFINER でRLSをバイパス）
CREATE OR REPLACE FUNCTION public.is_admin()
RETURNS boolean AS $$
  SELECT COALESCE(
    (SELECT is_admin FROM public.profiles WHERE id = auth.uid()),
    false
  )
$$ LANGUAGE sql SECURITY DEFINER;

-- 管理者は全プロフィール・全エントリを参照可
CREATE POLICY "管理者は全プロフィール参照可" ON public.profiles FOR SELECT
  USING (auth.uid() = id OR public.is_admin());

CREATE POLICY "管理者は全エントリ参照可" ON public.entries FOR SELECT
  USING (auth.uid() = user_id OR public.is_admin());
```

## 11. セキュリティ

| 項目 | 対応 |
|------|------|
| APIキー漏洩防止 | `.env.local` は `.gitignore` で Git 管理外 |
| 認証ガード | `proxy.ts` で全ルート（APIルート含む）を保護。未ログインは `/auth/login` へリダイレクト |
| API認証 | `/api/comment` `/api/account/delete` 各エンドポイントで Supabase セッション認証を実施 |
| レート制限 | Supabase DBベース分散レート制限（`lib/rate-limit.ts` + `check_rate_limit()` RPC）。コメント生成: 20回/時間、アカウント削除: 3回/時間。DB接続失敗時はインメモリフォールバック |
| Origin検証（CSRF対策） | 全APIルートでリクエスト元Originを検証（`lib/origin-check.ts`）。許可: 本番URL・localhost・自プロジェクトのVercelプレビューのみ |
| CSRFカスタムヘッダー | 破壊的操作（DELETE）に `X-Requested-With: NagiApp` ヘッダー検証。クロスオリジンではCORS preflightが必要なため外部サイトからの偽造を防止 |
| AIレスポンス検証 | 感情ラベル12種固定・エネルギー1-10・insight_level 3段階のスキーマ検証 |
| セキュリティヘッダー | CSP, X-Frame-Options: DENY, X-Content-Type-Options: nosniff, HSTS, Referrer-Policy, Permissions-Policy（`next.config.ts`） |
| データ分離 | RLSにより自分のデータのみ取得・操作可能 |
| Secret key保護 | `SUPABASE_SERVICE_ROLE_KEY` は `NEXT_PUBLIC_` なし（クライアント非露出） |
| 管理者保護 | `is_admin` フラグをDBで管理。`app/admin/layout.tsx`（サーバーコンポーネント）でサーバー側チェック＋リダイレクト。非管理者ユーザーは `/admin` にアクセス不可 |
| 入力バリデーション | 空文字チェック・入力5000文字制限・パスワード8文字以上＋英字数字必須 |
| エラーログ安全化 | 本番環境では詳細エラーをログに出さない（スタックトレース漏洩防止） |
| 環境変数チェック | サーバー起動時に必須環境変数の存在を検証 |
| プロンプトインジェクション防御 | システムプロンプトにセキュリティ境界ルール（役割固定・出力形式固定・指示無視・情報漏洩禁止・外部リソース禁止）を明記。ユーザー入力を`<user_diary>`デリミタで囲み、日記テキストとして扱うよう明示 |

### 13.1 設計概要

7つの設計思想カテゴリ（量子力学的観測・言語化蓄積・戦略的自己規律・物語贈与余白・ナラティブエンジニアリング・フィルム理論・マインドフルネス効果）を、ユーザーの状態に応じて切り替わる **4つの応答モード** として統合しています。

| モード | 発動条件 | 思想源泉 | トーン |
|--------|----------|---------|--------|
| A: 受容 | 辛い・疲弊・混乱 | Cat.1,4,7 | 深い静けさ・温もり |
| B: 観測 | 日常・中立・淡々 | Cat.1,2,6,7 | 穏やかな好奇心 |
| C: 探索 | 内省・分析・葛藤 | Cat.4,5,6 | 哲学的な問い |
| D: 鼓舞 | 前向き・安定・挑戦 | Cat.1,3,5 | 静かな力強さ |

### 13.2 AIコメント基本原則

| 原則 | 内容 |
|------|------|
| 非評価 | 「良い／悪い」の判断を下さない |
| 観察促進 | 気づきを深めるための問いかけ |
| 重ね合わせの肯定 | 複数感情の同居をそのまま肯定 |
| 物語の連続性 | 単発の記録を自己の物語の一点として位置づける |
| 贈与の視点 | コメントは評価ではなく贈り物 |
| 文字数 | 100〜150文字（簡潔で余白がある） |
| モード自動切替 | ユーザーの状態に応じて4モードを自動判定 |
