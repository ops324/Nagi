# Nagi ハーネス - Claude Code 運用ルール

最終更新: 2026-04-26
バージョン: v1.0.0

このファイルは Claude Code が起動時に自動ロードする運用ルール集。
Nagi プロジェクトにおける Claude Code の作業全般に適用される。

ハーネス全体像は `docs/harness-workflow.md` を参照。

---

## CC-1: 不確実な指摘・実装は「要確認」とマーク

確証がない情報、推測を含む情報、未検証の仮説を提示する場合は、
必ず明示的に「要確認」「未検証」「推測」のいずれかをマークする。

**悪い例**:
> このコードは Next.js 16 の推奨パターンに従っています。

**良い例**:
> このコードは Next.js 16 の推奨パターンに従っているはず(要確認: 公式ドキュメントの該当箇所未参照)。

---

## CC-2: フレームワーク仕様は Web 検索で公式確認

フレームワーク(Next.js、Supabase 等)、ライブラリの仕様・推奨パターン・
最新リリースノートに関わる判断は、Web 検索で公式ドキュメントを確認してから断定する。

特に以下のケースで必須:
- フレームワークのバージョン依存挙動を扱う時
- 「推奨パターン」「ベストプラクティス」を述べる時
- 古い記憶に基づいて実装する時

---

## CC-3: 動作確認チェックリストの全項目を Pass/Fail/Skip 報告

指示書に動作確認チェックリストが含まれる場合、**全項目**について
以下のいずれかを必ず報告する:

- **Pass**: 確認実施・期待通りの結果
- **Fail**: 確認実施・期待と異なる結果(詳細を記載)
- **Skip**: 何らかの理由で未実施(理由を必ず記載)

報告書冒頭に「動作確認結果サマリー: N/M Pass」を記載する。
1項目でも省略することは許容しない。Skip も含めて全項目を明示する。

**悪い例**:
> 動作確認: ビルド成功

**良い例**:
> 動作確認結果サマリー: 11/13 Pass, 1 Fail, 1 Skip
> - T10.1 ログイン機能: Pass
> - T10.2 ログアウト機能: Pass
> - T10.3 リダイレクト: Fail(/auth/login ではなく / にリダイレクトされた)
> - T10.4 ネットワーク確認: Skip(getUser 呼び出し回数の比較対象データなし)
> - (以下続く)

---

## CC-4: 「ビルド成功」と「動作確認完了」を区別して報告

TypeScript/Next.js のビルド成功は、動作の正常性を保証しない。
報告書では以下を**別項目**として記載する:

- **ビルド結果**: 成功 / 失敗(エラー詳細)
- **動作確認結果**: 上記 CC-3 のフォーマットで報告

**悪い例**:
> ビルド成功、動作 OK

**良い例**:
> ビルド結果: 成功(全14ページ生成)
> 動作確認結果サマリー: 13/13 Pass

---

## CC-5: バックアップブランチ作成有無を必ず報告

ファイル変更を伴うタスクでは、作業開始前にバックアップブランチを作成する。
作成有無を報告書に必ず記載する。

**ブランチ名規則**:
`backup/before-{タスク名}-{YYYYMMDD}`

**例**:
- `backup/before-harness-phase1-20260426`
- `backup/before-a3-security-md-update-20260427`

**報告例**:
> バックアップブランチ: `backup/before-harness-phase1-20260426` 作成済み

作成しなかった場合も「作成せず(理由: 〇〇)」と必ず明記する。
省略は禁止。

---

## CC-6: console.log 等の確認指示は実ランタイム確認を実施し結果報告

指示書に「console.log で確認せよ」「ブラウザで挙動確認せよ」等の指示があった場合、
**実際にランタイムで確認**し、結果を報告する。

**悪い例**:
> console.log を追加しました。

**良い例**:
> console.log を追加し、開発サーバーで `/account` ページにアクセスして確認。
> 結果: `claims.email` に `user@example.com` が出力された。
> スクリーンショット添付不可のため、出力ログをそのまま貼り付け:
> ```
> [SERVER] claims: { sub: "abc-123", email: "user@example.com", ... }
> ```

確認できなかった場合は「確認不可(理由: 〇〇)」と明記する。

---

## CC-7: ファイル更新時は差分サマリーを報告

ファイル更新を行った場合、以下を必ず報告する:

- **対象ファイル名**(フルパス)
- **更新前バージョン → 更新後バージョン**(該当する場合)
- **どのセクションを変更したか**(セクション名 or 行番号範囲)
- **変更の要約**(1〜3行)

バージョン番号だけの報告は禁止。

**悪い例**:
> docs/仕様書.md を v1.31.0 → v1.32.0 に更新

**良い例**:
> docs/仕様書.md を v1.31.0 → v1.32.0 に更新
> - セクション 3.2「認証フロー」: A-2α 統合内容を追記(L450-L475)
> - セクション 7「既知の制約」: 「マウント時に一瞬コンテンツが見える可能性」を追加(L890)
> - 改訂履歴: v1.32.0 のエントリを追加(L1200)

## 適用範囲

これらのルールは Nagi プロジェクト内の Claude Code 作業**全般**に適用される。
特定のファイル種別・ディレクトリに限定しない。

ハーネス改訂が必要な場合は、`docs/harness-workflow.md` の改訂手順に従う。
