# FF: 機能・デザイン着手フロー（着手時ゲート）

最終更新: 2026-06-25
バージョン: v1.0.0

新機能・デザイン・UI変更に着手したら、**コードを書く前に**以下を宣言する。
このゲートの目的はスパゲッティコード（重複実装・コンポーネント肥大化・規約逸脱）を
設計時に防ぐこと。省略・後回しは禁止（CC-3 と同じ強制度で扱う）。

着手時に FF-1〜FF-5 を箇条書きで宣言してから実装に入ること。

---

## FF-1: 影響範囲の宣言

実装に入る前に、変更の波及を明示する。

- **触るファイル**／**触らないファイル**を列挙する
- 既存の state・型（`app/types.ts`）・API ルートへの波及を 1〜3 行で述べる
- 不明な依存があれば grep / 参照検索で確認してから宣言する

---

## FF-2: 再利用の優先（スパゲッティ第一防壁）

新規作成の前に、既存資産で済まないかを必ず確認する。

- 似た既存コンポーネント・関数・フックを grep で探したか（**探索した語と結果を記載**）
- `app/components/ui/` の既存プリミティブ（`SearchBar` `TabBar` 等）で足りないか確認したか
- それでも新規作成する場合、**「なぜ既存を使えないか」を明記**する
- コピー&ペーストでの分岐実装を避け、共通化できる箇所は関数/コンポーネントに寄せる

---

## FF-3: 肥大化ガード

- 1 コンポーネントが概ね **200 行**を超えそうなら、書く前に分割単位を決める
- `HomeClient.tsx` に state や JSX を足す場合、子コンポーネント化を検討する
  （v1.65〜v1.73 で 1236 行を 8 PR かけて解体した教訓。再肥大化させない）
- 「1 部品 ＝ 1 責務」を基準に、責務が増えるなら分けてから足す

---

## FF-4: 規約セルフチェック

`.claude/rules/` の既存規約のうち、毎回踏みやすい項目を着手時に自己点検する。

- 型は `app/types.ts` に集約したか（コンポーネント内に型を散らさない）
- DB 取得は snake_case → camelCase を**明示変換**したか／INSERT は camelCase → snake_case か
- 色・余白はデザイントークン（`app/globals.css` の CSS 変数）を使い、**ハードコードしていない**か
- `"use client"` は必要な箇所だけか（データフェッチはサーバー優先）

---

## FF-5: セキュリティの床（該当時のみ）

以下のいずれかに触れたか？

- 認証 / RLS ポリシー / `ANTHROPIC_API_KEY` / `SUPABASE_SERVICE_ROLE_KEY`
- 環境変数の新規参照 / `createAdminClient` / API ルートのデータ操作

→ **1 つでも該当するなら「PR 前に `/security-review` 必須」と宣言する**（CC-9 と連動）。
該当しない場合も「FF-5: 非該当」と明示する。

---

## 機械ガードとの関係

- このゲート（FF）は**設計時の人/AI 判断**を担う（機械では検出できない領域）。
- **型崩れ・lint 違反**は応答終了時の Stop hook（`eslint` + `tsc --noEmit`）が自動で落とす。
- **build / test / e2e と `/review`** は PR 前に CC-9 で実行する。
- 三層（FF＝設計時 / hook＝毎ターン機械 / CC-9＝PR前）で、どこかが抜けても次層で拾う。
